# ADR-0002：状态机变更与回归测试基线

- 状态：Accepted
- 日期：2026-05-16
- 适用范围：`java/timer-backend` 状态机逻辑、`cpp/plc-collector` 事件输入、`frontend/station-screen` 状态展示、发布与运维验证

## 背景

当前系统以 Java 后端状态机作为业务中枢：消费 Redis Stream 事件后，决定工位状态、告警、推送与落库行为。  
状态机规则一旦变更，会同时影响：

- 现场屏幕显示准确性（WebSocket 推送结果）
- 告警触发与解除时机
- 生产节拍（CT）统计与审计可追溯性

此前缺少统一“怎么改、怎么测、怎么发布”的基线，容易出现：

- 规则改动后仅做单点验证，未覆盖边界路径
- 环境间结果不一致（本地通过、线上异常）
- 回滚时缺乏可复现输入集

## 决策

采用“**状态机变更必须伴随转换表变更 + 最小回归集 + 灰度观察窗口**”策略。

## 变更规范

### 1) 先文档后代码

任何状态机规则调整，必须先更新或新增文档（建议位置：`docs/decisions` 或状态机专用说明），至少包含：

- 变更前后状态转换差异
- 触发条件（事件类型、字段条件、时间窗口）
- 异常路径处理（超时、乱序、重复事件、未知事件）

### 2) 转换表为唯一事实来源

状态机以“状态 + 事件 -> 新状态 + 副作用”表达，副作用至少标注：

- 是否推送 WebSocket
- 是否写 Oracle
- 是否触发告警
- 是否更新 Redis 状态缓存

不允许仅通过散落代码分支表达关键规则而无对应转换表说明。

### 3) 破坏性规则变更要求

若变更会影响历史口径（CT 统计、告警口径、审计解释），视为破坏性变更，必须：

1. 在 ADR 中单独说明影响面与迁移方案。
2. 提供“切换前后对照样例”。
3. 明确回滚开关或回滚步骤。

## 回归测试基线

每次状态机变更至少覆盖以下 8 类场景：

1. **标准流程**：正常开始 -> 运行 -> 完成  
2. **重复事件**：同一事件重复输入不应导致状态跳变污染  
3. **乱序事件**：先到后发场景需有可解释处理  
4. **缺失事件**：关键事件缺失时进入可恢复或告警态  
5. **超时场景**：超时触发告警与恢复路径正确  
6. **异常码流程**：`abnormalCode` 触发告警并可解除  
7. **跨班次/重启恢复**：服务重启后状态恢复策略一致  
8. **高频输入**：短时高频事件不导致状态抖动或性能退化

## 发布与灰度要求

状态机变更发布需执行以下流程：

1. 本地/测试环境使用固定样本回放（覆盖上述 8 类场景）。
2. 预发布环境观察至少一个完整业务时段（含开始与结束）。
3. 生产灰度时开启重点指标观察：
   - 状态切换频次异常
   - 告警数量异常波动
   - WebSocket 推送失败率
   - 后端错误日志与事件丢弃量
4. 达到观察窗口后再全量。

## 回滚策略

若上线后出现状态异常或告警误触发：

1. 回滚到上一版本状态机规则（代码或配置开关）。
2. 保留异常时段事件样本，执行离线回放复盘。
3. 对比“预期状态轨迹 vs 实际轨迹”，补齐测试样例后再发版。

## 验收标准

- 状态机变更 PR 附带转换表变更说明与最小回归结果。
- 回归样例可在本地与 CI 重复执行，结果一致。
- 灰度期间无关键状态误判、无异常告警风暴、无前端核心展示错误。
