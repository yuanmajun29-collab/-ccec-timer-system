# ADR-0003：告警分级与抑制策略

- 状态：Accepted
- 日期：2026-05-16
- 适用范围：`java/timer-backend` 告警逻辑、`frontend/admin-console` 告警展示、监控告警系统（Prometheus/Alertmanager）、运维值守流程

## 背景

当前系统告警来源包括设备异常、状态机超时、链路故障与服务健康异常。  
若告警未分级或缺少抑制机制，容易出现两类问题：

- **漏报**：关键故障未被及时处理，影响现场节拍与生产连续性。
- **告警风暴**：同一故障持续重复触发，掩盖真正高优先级问题。

需要统一“告警等级、触发条件、抑制策略、恢复通知、升级路径”。

## 决策

采用“**四级告警 + 去抖抑制 + 恢复闭环 + 可审计追踪**”策略。

## 告警分级模型

### P1（紧急）

- 定义：直接影响生产连续性或系统不可用。
- 示例：后端不可用、Redis/Oracle 不可用、核心状态机失效、关键工位全量失联。
- 处理要求：立即通知值班人员，要求实时介入。

### P2（高）

- 定义：局部业务严重异常，短时间可能升级为生产风险。
- 示例：单工位持续异常码、连续超时、MQTT/采集链路不稳定。
- 处理要求：限定时间内处置，需跟踪到恢复。

### P3（中）

- 定义：功能退化或可绕过异常，不立即影响主流程。
- 示例：管理端非核心功能异常、单项指标偏离阈值。
- 处理要求：纳入当班处理或下一工作窗口修复。

### P4（低）

- 定义：提示类或优化类信息。
- 示例：可自动恢复的瞬时抖动、轻微性能波动。
- 处理要求：用于观察与趋势分析，不触发紧急通知。

## 告警抑制与去抖策略

### 1) 时间去抖（Debounce）

- 同一告警键在短时间内重复出现，仅记录计数，不重复通知。
- 默认建议：
  - P1：30 秒窗口
  - P2：60 秒窗口
  - P3/P4：120 秒窗口

### 2) 合并策略（Aggregation）

- 同类告警按“站位 + 告警码 + 来源组件”聚合。
- 通知内容包含：
  - 首次发生时间
  - 最近发生时间
  - 累计次数
  - 当前状态（激活/恢复）

### 3) 升级策略（Escalation）

- 若同一 P2 告警在观察窗口内持续未恢复，自动升级为 P1。
- 升级动作必须产生日志与审计记录，避免静默升级。

### 4) 恢复通知（Recovery）

- 告警恢复时必须发送“恢复事件”，与触发事件形成闭环。
- 恢复通知需附带持续时长与影响范围，支持复盘。

## 告警键与幂等规范

- 告警唯一键建议：`{source}:{station}:{alarmCode}:{version}`。
- 相同键在激活期间重复触发应幂等处理（更新计数和最后触发时间，不重复建新告警单）。
- 告警状态至少包含：`ACTIVE`、`ACKED`、`RESOLVED`。

## 数据与审计要求

- Oracle 侧应保留告警生命周期记录（触发、确认、恢复、升级）。
- 管理端需要支持按时间、站位、等级、状态检索。
- 审计日志应包含操作者、动作类型、时间、备注（确认/静音/恢复标注）。

## 发布与验证基线

每次告警策略调整，至少验证：

1. 同一异常连续触发时不会造成通知风暴。
2. P1/P2 告警在规定窗口内可达并可追踪。
3. 告警恢复后能正确闭环并更新状态。
4. 升级规则触发条件准确，不误升级。
5. 管理端筛选结果与数据库状态一致。

## 回滚策略

若上线后出现漏报或误报显著增加：

1. 回滚至上一版本阈值与抑制参数。
2. 暂时关闭自动升级策略（仅保留原始分级）。
3. 导出异常时间段告警样本，进行阈值与规则复盘。
4. 修正后在预发环境重新压测再上线。

## 验收标准

- 告警风暴场景下通知量显著下降，但关键告警可达性不下降。
- 每条 P1/P2 告警均具备触发、处理、恢复的完整生命周期记录。
- 值班人员可在管理端快速定位“当前未恢复的高优先级告警”。
