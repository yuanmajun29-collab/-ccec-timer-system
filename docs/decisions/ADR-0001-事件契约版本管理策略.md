# ADR-0001：事件契约版本管理策略

- 状态：Accepted
- 日期：2026-05-16
- 适用范围：`cpp/plc-collector`、`java/timer-backend`、`frontend/station-screen`、`frontend/admin-console`、运维与排障链路

## 背景

当前系统的核心链路为：

`PLC 采集 -> Redis Stream(payload) -> Java 状态机 -> WebSocket 推送 -> 工位屏/管理端`

其中 `StationEvent` JSON 是跨语言、跨进程、跨团队的共享契约。一旦字段语义或结构发生不兼容变化，将直接影响：

- 后端消费与状态机判断
- WebSocket 推送内容与前端展示
- 线上回放、排障与审计解释一致性

项目已具备多运行环境（本地、容器、生产增强），需要明确一套可执行的版本策略，避免“同名字段不同语义”或“静默破坏兼容”的风险。

## 决策

采用“**显式版本字段 + 向后兼容优先 + 破坏性变更走双轨迁移**”策略。

### 1) 事件版本字段

- 每条 `StationEvent` 增加（或保留）`eventVersion` 字段，使用语义化主次版本：
  - `1.x`：兼容演进（新增可选字段、枚举扩展等）
  - `2.0`：不兼容变更（字段删除、语义重定义、必填约束变化）
- `eventVersion` 必须由采集端写入，不允许后端推断。

### 2) 兼容性原则

- **新增字段**：默认仅允许新增“可选字段”，且旧消费者可忽略。
- **删除/改名字段**：视为破坏性变更，禁止直接上线。
- **字段语义变化**：即使名字不变，也按破坏性变更处理。
- **枚举扩展**：消费者必须提供未知值兜底，不得因新枚举直接失败。

### 3) 双轨迁移（破坏性变更）

当出现 `1.x -> 2.0` 的不兼容升级时，执行双轨窗口：

1. 采集端可配置输出 `v1`/`v2`（或同时输出，按流/字段区分）。
2. 后端在窗口期同时支持 `v1` 与 `v2` 解析。
3. 前端与下游消费者完成切换后，再移除旧版本支持。
4. 移除旧版本前，必须完成回放验证与生产灰度观察。

### 4) 契约文档与变更门禁

- 事件字段字典必须落地在文档并标记版本（建议后续新增 `docs/event-contract.md`）。
- 任何涉及事件契约的 PR，必须包含：
  - 变更类型（兼容/不兼容）
  - 影响面（C++/Java/前端/运维）
  - 回滚方案
  - 最小验证清单

## 落地步骤（本项目）

1. 在 C++ 采集端统一输出 `eventVersion`（默认 `1.0`）。
2. 在 Java 后端消费入口增加版本分派与未知版本告警日志。
3. 在前端 WebSocket 解析层增加未知字段与未知枚举兜底。
4. 补充契约文档，明确每个字段：
   - 名称、类型、是否必填、单位、取值范围、示例、引入版本、废弃版本
5. CI 增加契约回归检查：
   - 至少一组 `v1` 样例事件持续可回放通过
   - 新增版本样例必须包含正向与异常路径

## 影响与权衡

### 正向收益

- 降低跨语言改动导致的线上故障概率。
- 升级与回滚路径清晰，便于灰度发布。
- 排障时可快速定位“数据版本”和“解析路径”。

### 成本与代价

- 破坏性升级期间需要维护双版本解析，短期复杂度上升。
- 文档与样例测试需要持续维护。

## 不采纳方案

- **方案A：不加版本字段，依赖字段存在性推断**
  - 问题：高歧义，排障困难，容易出现误判。
- **方案B：每次变更都新建独立 Stream**
  - 问题：运维与消费链路复杂度显著增加，不适合当前阶段。

## 回滚策略

若新版本上线后出现异常：

1. 采集端配置回切为旧版本事件输出（`v1`）。
2. 后端保持双轨解析能力，优先消费旧版本。
3. 前端继续按旧字段渲染，同时记录新版本异常样本。
4. 通过 Redis 样本回放复盘后再重启升级。

## 验收标准

- 新旧版本事件均可被后端解析且状态机结果一致（预期范围内）。
- 工位屏与管理端在版本切换期间无白屏、无关键字段缺失崩溃。
- 生产日志中可按 `eventVersion` 定位问题，且告警可区分版本来源。
